# 診断モジュール規約

登録情報(生年月日・氏名・住所など)を基にした診断を、コア改修なしで追加・保留・削除できるようにするための共通規約。新しい診断を追加するときは必ず本規約に従う。

---

## 1. 基本思想

- 各診断は独立した**診断モジュール**として実装する。モジュールの追加・無効化がコア(登録フロー・マイタイプ画面・バッチ・Flex カード)の改修を要求してはならない
- 診断の計算はすべて決定的。LLM は計算に関与しない(CLAUDE.md ルール 1)
- 一度リリースしたモジュールのロジック・固定値は変更しない(同ルール 2)。方式を変えたい場合は新バージョンのモジュールとして追加し、旧結果と区別する

## 2. モジュールインターフェース

```typescript
interface DiagnosisModule {
  id: string;                 // 例: "zodiac", "numerology_lifepath"
  version: number;            // ロジック改版時にインクリメント
  requiredInputs: InputKey[]; // このモジュールが必須とする登録情報
  optionalInputs: InputKey[]; // あれば精度が上がる情報(例: birth_time)
  clientSafe: boolean;        // 端末内のみで計算可能か(友達診断で使用可か)
  compute(inputs: ProfileInputs, masters: MasterData): ResultJson;
}
```

### 入力カタログ(InputKey)

| キー | 内容 | 収集タイミング |
|------|------|---------------|
| `birth_date` | 生年月日(西暦のみ) | 初回登録(必須) |
| `birth_time` | 出生時刻 | 初回登録(任意) |
| `name_kana` / `name_romaji` | 氏名かな/確定ローマ字 | 初回登録 |
| `name_kanji` | 氏名漢字 | **保留中モジュール専用のため現在は収集しない** |
| `home_latlng` | 自宅座標 | 初回登録(方位系のみ使用) |

**登録フローの入力項目は「有効なモジュールの requiredInputs の和集合」から自動導出する。** どのモジュールも要求しない情報は収集しない(個人情報最小化)。将来のモジュールが新しい入力を要求する場合、既存ユーザーには「追加情報を入力すると新しい診断が解放される」導線で追加収集する。

## 3. モジュールステータス

| ステータス | 意味 | 扱い |
|-----------|------|------|
| `enabled` | 稼働中 | 登録時計算・マイタイプ表示・LLM 文脈への提供 |
| `held` | **保留** | 仕様書は保持。実装・マスタ整備・入力収集を行わない。コードが存在しても本番で有効化しない |
| `planned` | 構想 | 仕様書のみ |

ステータスは設定(module_config)で管理し、コードのコメントアウトで表現しない。

## 4. 結果の保存

- `diag_results(user_id, module_id, module_version, result_json, computed_at)` に**モジュールごとに 1 行**で保存する。固定カラムを増やさない
- モジュール追加・保留解除時は、対象モジュールのみ全ユーザー再計算するバッチを流す(他モジュールの既存結果には触れない)

## 5. 表示・生成への接続

- マイタイプ画面と Flex タイプカードは、enabled モジュールの結果を定義順に描画する(セクションの動的生成)
- 各モジュールの説明文は固定コピーとしてコンテンツマスタに持つ(`master_module_contents(module_id, key, text)`)
- 日次運勢の LLM プロンプトへは、モジュールが提供する構造化サマリのみを渡す。プロンプトテンプレートはモジュール一覧をループで注入し、モジュール追加時にテンプレート改修を不要にする

## 6. 追加手順(チェックリスト)

1. 仕様書を docs/ に追加(算出方式・流派差の論点・既定値を明記)
2. 未確定論点を人間が確定(既定値承認を含む)
3. logic-dev がモジュール実装+ユニットテスト(検算例必須)
4. 必要なマスタデータを整備・検証
5. コンテンツマスタ(固定コピー)を投入
6. qa-lead のレビュー+品質ゲート
7. module_config で enabled 化 → 再計算バッチ → リリース
