## 6. Phase 3 概要

設計書 §8 Phase 3 = 「キャラデザイン適用、ハイブリッド表示の作り込み、応答メッセージ体験、招待運用開始 → クローズドβ開始」。デザインの一次情報は docs/06(UIデザイン指示書)・docs/07(デザインハンドオフ。今日のページのビジュアルの正)。Phase 3 実装は両者に従う。

Phase 3 の実装量の大半は「作り込み=デザイン品質・レイアウト洗練・モーション」であり、機能追加より品質向上に軸足がある。制作物(HEX・リッチメニュー画像)待ちでブロックされる部分と、それを待たずに組める基盤(トークン注入機構・レイアウト土台・モーション実装)を分離して着手順を最適化する。

## 9. 依存関係・クリティカルパス

### 今すぐ着手可能(制作物待ちにブロックされない基盤)
- **3-1 キャラテーマ トークン基盤**(プレースホルダ HEX で先行。P-D 確認と並行)
- **3-3 ハイブリッド表示**(3-1 完了後すぐ。色確定不要、レイアウト作業)
- **3-5 のコピー/構造部分**(色反映を除く。UIライティング調整・③カード構造)
- **3-7 キャラアセット確認**(独立)

### 制作物・人間確認待ちでブロックされる実装
- **3-2 キャラデザイン適用** ← P-A(HEX 確定)
- **3-4 リッチメニュー設定** ← P-B(画像制作)
- **3-5 の色反映** ← P-A
- **3-6 モーション** ← P-C(範囲判断)

### クリティカルパス
```
P-D(トークン抽出) ─┐
                    ├→ 3-1 基盤 ─→ 3-2 デザイン適用 ─→ 3-Q/3-E ─→ (実機 C5) ─→ 3-8 β開始
P-A(HEX 確定) ──────┘        (並行: 3-3 / 3-5 / 3-6)
P-B(RM 画像) ─→ 3-4 ────────────────────────────────┘
```
最長経路 = **P-A(HEX 確定)→ 3-1 → 3-2 → 品質ゲート → 実機テスト → β開始**。
- クリティカルパスの起点は **P-A(HEX 確定)と P-D(トークン抽出)**。この 2 つを最優先で解消すると後続がほぐれる。
- 3-1 は P-A を待たずプレースホルダで着手し、P-A 確定後に 3-2 で実値差替 → HEX 待ちの遊休を最小化する。
- P-B(リッチメニュー画像)は独立した並行経路。早期に制作依頼を出せば 3-4 がクリティカルパス外に留まる。

## 10. Phase 3 受入判定基準(Definition of Done)

Phase 3 完了 = 以下を全て充足:
1. 実装タスク 3-1〜3-7 が完了し、各 DoD を満たす
2. 3-Q 品質ゲート合格(Blocker/High 指摘ゼロ)、3-E E2E green・既存回帰なし
3. `packages/engine`/`api`/`batch` のユニットテスト全 green(色・トークン変更による回帰なし)
4. docs/06 品質基準(タップ領域 44px・コントラスト AA・reduced-motion・初回表示 3 秒・セーフエリア)を全画面で満たす
5. docs/09 の Phase 3 追加項目を含む実機テストに合格(人間実施)

### クローズドβ開始ゲート(4 点が全て揃うこと)
| 観点 | 条件 |

| 実装 | Phase 3 の DoD(上記 1〜4)充足。qa-lead ゲート合格 + pm 受入判定 |
