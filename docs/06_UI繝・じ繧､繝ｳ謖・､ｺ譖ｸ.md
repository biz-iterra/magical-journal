# UIデザイン指示書 — マジカルジャーナル

この文書は、Claude(Claude Code)にマジカルジャーナルの UI デザインを依頼するための指示書である。packages/liff の実装前に本指示書に従ってデザインを確定させること。CLAUDE.md と docs/01〜05 を前提知識として必ず読み込むこと。

---

## 1. あなたの役割と進め方

あなたは小規模スタジオのデザインリードとして振る舞う。テンプレート的な出力ではなく、このプロダクト固有の根拠ある選択をすること。

**必ず3フェーズで進め、各フェーズの終わりに承認を得るまで次へ進まない。**

| フェーズ | 内容 | 成果物 |
|---------|------|--------|
| A. デザイン計画 | トークン案(カラー・タイプ・レイアウト・シグネチャ)+主要画面のワイヤー(ASCII可)を提示。自分の案が「どんな占いアプリにも出せる一般解」になっていないか自己批評し、修正点を明記する | デザイン計画書(markdown) |
| B. プロトタイプ | 「今日のページ」1画面のみ HTML プロトタイプで実装(キャラテーマ切替のデモ含む) | 単一 HTML |
| C. 全画面展開 | 承認されたトークンで vanilla-extract テーマ+全画面実装 | packages/liff |

---

## 2. プロダクトの核(デザインの出発点)

- **これは占い結果表示アプリではなく「毎日開く手帖(ジャーナル)」である。** 派手さより、毎朝開いて心が整う静かな心地よさを優先する。ダッシュボードではなく暦・手帖のメタファーで考えること
- 主役は 12 種のキャラクター。各キャラは自然モチーフ(光・月・風・霧・炎・滝・山・岩・虹・露・朝陽・湖)とテーマカラーを持つ(docs/05 とキャラ YAML が一次定義)
- 利用者: クローズド招待者。LINE に慣れた一般の大人。スピリチュアルに強い関心がある層とは限らないため、**過度にスピリチュアルな装飾は忌避される**
- 各画面の仕事: 今日のページ=「今日をどう過ごすかが 10 秒で掴める」/ マイタイプ=「自分のキャラに愛着が湧く」/ 友達診断=「入力 30 秒で盛り上がれる」

## 3. 固定条件(変更不可)

- 実行環境: LINE の LIFF(WebView)。**モバイル縦画面専用**。PC 対応は不要
- 技術: React + Vite + TypeScript + vanilla-extract。CSS はすべてトークン経由
- キャラのテーマカラー・差し色は docs/05 / キャラ YAML の定義値を使う。**独自の色を発明しない**
- 方位マップ: Google Maps 上に扇形オーバーレイ(四正30度・四隅60度)。吉=緑系・凶=赤系の意味色は固定
- Flex Message カード(運勢カード・タイプカード)にも同じ世界観を適用する。Flex の表現制約(限定的なレイアウト・色指定)内で成立するトークン設計にすること

## 4. デザイン方針

### 世界観の軸

「自然の気配を借りた、静かで上質な暦」。キャラの自然モチーフ(月・霧・湖…)が画面の空気をつくる。装飾で飾るのではなく、余白・色・光のグラデーションで気配を表現する。

### 動的キャラテーマ(本アプリ最大の設計課題)

- ベースは中立で落ち着いた基調(紙・墨のようなニュートラル)とし、**ユーザーのキャラのテーマカラーがアクセントとして画面全体の空気を変える**構造にする
- 12 色すべてで成立するトークン設計(CSS カスタムプロパティでキャラテーマを注入)。淡い色(露・光)でも濃い色(月・滝)でもコントラスト基準を満たすこと。必要なら各キャラに on-color(文字色)トークンを定義する
- ハイブリッドタイプは主キャラ(大)+副キャラ(小)の並列表示。テーマカラーは主キャラを採用する

### やってはいけないこと

- 占いアプリの常套句: 紫グラデーション、星屑・キラキラの多用、水晶玉、ゴシックホラー調
- 生成 AI 的な一般解: クリーム背景+セリフ大見出し+テラコッタ系アクセント / 黒背景+蛍光単色アクセント / 罫線だらけの新聞風。これらは「選択」ではなく「デフォルト」であり不採用
- 過剰な角丸+影+絵文字のポップ UI(子ども向けに見せない)
- 意味のない装飾番号・ダミーの統計値・目的のないアニメーション

### シグネチャ

このアプリが記憶される要素を 1 つだけ選び、そこに大胆さを集中させる(例: キャラモチーフが息づく今日のページのヘッダー表現、方位盤の独自ビジュアルなど。候補を A フェーズで 2〜3 案提示)。それ以外は静かに規律を保つ。

## 5. タイポグラフィ

- 日本語が主言語。**書体選択がこのデザインの人格を決める**。手帖・暦の世界観に合う組み合わせを意図を持って選ぶこと(例の方向性: 見出し=明朝系(Shippori Mincho / Zen Old Mincho 等)、本文=可読性の高いゴシック系。ただしこれは例であり、A フェーズで根拠とともに提案する)
- Google Fonts の日本語ウェブフォントはサブセット・`font-display: swap` で読み込み、LIFF 初回表示を遅らせない
- 数字(方位角度・日付・星番号)は等幅または表組み用数字で揃える
- 明確なタイプスケールを定義し、トークン化する

## 6. 画面別の要点

| 画面 | デザイン上の要点 |
|------|-----------------|
| 初回登録 | 3ステップ(生年月日→住所→キャラスタイル選択)。キャラスタイル選択は男女キャラを対等に並べ、診断の山場として演出する |
| 今日のページ | 最重要画面。上から「今日の日付と暦情報 → 運勢文(キャラが語る) → 吉凶方位サマリ → 方位マップ」。運勢文はキャラのアイコン+吹き出し等で「キャラが語っている」ことが伝わる形式 |
| 方位マップ | 地図タイルの上でも吉凶が瞬時に判別できるオーバーレイの透明度・彩度設計。凡例必須。色覚多様性に配慮し、色だけに頼らず記号(○/△/×等)を併用 |
| マイタイプ詳細 | キャラ画像を主役に、3軸(星座・ポテンシャル・本命星)を整理して提示。ハイブリッドは主大+副小 |
| 友達診断 | 未登録者も使う入口。入力は最小(生年月日+任意の時刻)、結果はスクリーンショットして共有したくなるカード型に |
| 月間 | 今日のページの月版。カレンダー的な一覧性を優先 |
| 設定 | 装飾不要。明快さのみ |

## 7. モーション

- 使うのは最大 2 箇所: (1) 今日のページ初回表示の静かな立ち上がり、(2) 診断結果の開示演出(登録完了・友達診断)。それ以外のアニメーションは原則なし
- `prefers-reduced-motion` を必ず尊重する

## 8. UIライティング

- 文言はデザインの一部。ボタンは動作をそのまま書く(「保存する」「診断する」)。曖昧な「送信」「OK」を使わない
- キャラの語り(運勢文)と UI の地の文を明確に区別する。UI の地の文は敬体で簡潔、キャラ口調を混ぜない
- エラーは原因と次の行動を書く。謝罪文・曖昧表現は不可。空状態(運勢未生成等)は次の行動への案内にする
- 占い結果の断定表現は避け、「〜に向く日」「〜がおすすめ」のトーンで統一する

## 9. 品質基準(暗黙の床)

- タップ領域 44px 以上、コントラスト WCAG AA、キーボードフォーカス可視
- 中位スペックの Android WebView で初回表示 3 秒以内(フォント・キャラ画像の最適化を含む)
- v1 はライトテーマのみ。ただしトークン構造はダークテーマを後から追加できる形にする
- セーフエリア(ノッチ・LIFF ヘッダー)を考慮したレイアウト

## 10. Aフェーズの提出フォーマット

1. 主題の一文定義(このアプリは何で、誰の、どんな 10 秒のために存在するか)
2. カラートークン案(ニュートラル基調 4〜6 色の hex+キャラテーマ注入の仕組み)
3. タイプ案(見出し/本文/数字の 3 役、選定理由付き)
4. 今日のページのワイヤー 2 案(ASCII)+シグネチャ候補 2〜3 案
5. 自己批評: 「一般的な占いアプリのデフォルト」との差分を明記し、デフォルトに落ちていた箇所の修正内容を書く

承認後に B フェーズへ進むこと。
