# 13. デプロイ・接続手順書(運用開始ランブック)

実装完了(Phase 1〜2)後、**本番として動かすまでの接続・設定作業**を手順化する。上から順に実施すればクローズドβを開始できる。各手順に検証(✔)を付けているので、通ってから次へ進む。

> このドキュメントは**人間が実施する運用手順**。コードは GitHub `main`(https://github.com/biz-iterra/magical-journal)に全て入っている。

---

## 全体像(3つの設置先)

| 設置先 | 動かすもの | 設定する場所 |
|--------|-----------|-------------|
| **NAS(UGREEN DXP4800GT)** | api / batch / tunnel(Docker) | `infra/.env` |
| **Cloudflare Workers** | LIFF フロント(静的 SPA) | ダッシュボードのビルド環境変数 |
| **外部サービス** | LINE Developers / Google Cloud | 各コンソール |

公開 URL(確定済み): API = `https://mj-api.iterra.online`(iterra.online を Cloudflare Tunnel で公開)。

## 事前に手元に用意するもの(取得元)

| 値 | 取得元 |
|----|--------|
| LINE Login チャネル ID | LINE Developers → 対象 LINE Login チャネル → チャネル基本設定「チャネル ID」 |
| LIFF ID | LINE Developers → LINE Login チャネル → LIFF タブ |

## A. NAS バックエンド(api / batch / tunnel)

NAS に SSH で入る(`ssh <ユーザー>@<NASのIP>`)。作業ディレクトリは `/volume1/docker/magical-journal`。

### A-1. 最新コードを取得(git 未インストールのため tar)

```
curl -L https://github.com/biz-iterra/magical-journal/archive/refs/heads/main.tar.gz | tar xz --strip-components=1 -C /volume1/docker/magical-journal
```

✔ `ls /volume1/docker/magical-journal/infra` に `docker-compose.yml` があること。

### A-2. `.env` を作成・記入

テンプレートを `infra/.env` にコピーして値を埋める(**リポジトリ内のファイルには書かない**。コミット事故防止)。

```
cd /volume1/docker/magical-journal/infra && cp .env.example .env && vi .env
```

記入する値(この手順書の各設置先で使う値を集約):

```env
# LINE Messaging API(Webhook 署名検証・応答)
LINE_CHANNEL_SECRET=＜Messaging API チャネルシークレット＞
LINE_CHANNEL_ACCESS_TOKEN=＜Messaging API 長期アクセストークン＞

# LINE Login(LIFF トークン検証・Flex ディープリンク)
LINE_LOGIN_CHANNEL_ID=＜LINE Login チャネル ID＞
LIFF_ID=＜LIFF ID＞

# Flex カードのキャラ画像を配信するフロントの origin(末尾スラッシュなし)
PUBLIC_ASSET_BASE_URL=https://magical-journal.＜アカウント＞.workers.dev

# Cloudflare Tunnel
CLOUDFLARE_TUNNEL_TOKEN=＜Tunnel トークン＞

# LLM(夜間・月次バッチの文章生成)
LLM_PROVIDER=claude          # または openai
ANTHROPIC_API_KEY=＜Claude キー＞     # LLM_PROVIDER=claude のとき
OPENAI_API_KEY=＜OpenAI キー＞        # LLM_PROVIDER=openai のとき
# CLAUDE_MODEL / OPENAI_MODEL は既定(claude-sonnet-5 / gpt-5)を使うなら未設定でよい

# 招待リスト(クローズド運用)★重要
ALLOWED_LINE_USER_IDS=＜自分の LINE ユーザー ID＞    # カンマ区切りで複数可
```

> ★ **`ALLOWED_LINE_USER_IDS` を必ず設定すること。** 本番(`NODE_ENV=production`)で空だと、セキュリティのため**全リクエストを 403 で拒否(フェイルクローズ)**する。最低でも自分の ID を入れる。招待者が増えたらカンマ区切りで追加。
>
> `GOOGLE_MAPS_API_KEY` は NAS では不要(地図・住所検索はフロント側で使うため、後述の Cloudflare 環境変数に設定する)。

保存: `vi` は `Esc` → `:wq`。**または** `vi` が不安なら次のヒアドキュメントで一括作成:
```
cat > /volume1/docker/magical-journal/infra/.env <<'EOF'
LINE_CHANNEL_SECRET=...
（上記の内容を全部貼る）
EOF
```

### A-3. ビルド & 起動

```
cd /volume1/docker/magical-journal/infra
sudo docker compose build api batch
sudo docker compose up -d api batch tunnel
```

> `sudo` はお使いのユーザーが docker グループなら不要。DB(SQLite)は Docker ボリューム `db-data` に永続化され、テーブルは api 起動時に自動作成される(マイグレーション不要)。

✔ 疎通:
```
curl -s http://localhost:3000/api/health          # {"status":"ok"}
sudo docker compose ps                              # api/batch/tunnel が Up
sudo docker compose logs --tail 20 tunnel           # "Registered tunnel connection" が出る
```

### A-4. 公開 URL の確認(トンネル経由)

ブラウザ or 別端末から:
```
curl -s https://mj-api.iterra.online/api/health     # {"status":"ok"}
```
✔ 200 が返れば NAS 公開まで完了。返らない場合は Cloudflare 側の Public Hostname 設定(Service = `http://api:3000`)と、iterra.online の DNS が Cloudflare 管理(Active)であることを確認。

## B. フロント(Cloudflare Workers)

Cloudflare ダッシュボード → Workers & Pages → `magical-journal` プロジェクト。

### B-1. ビルド環境変数を設定

設定 → ビルド → 環境変数 に以下を追加:

| 変数名 | 値 |
|--------|-----|
| `VITE_API_BASE_URL` | `https://mj-api.iterra.online` |
| `VITE_LIFF_ID` | ＜LIFF ID＞ |
| `VITE_GOOGLE_MAPS_API_KEY` | ＜Google Maps API キー＞ |
| `NODE_VERSION` | `22` |

### B-2. 再デプロイ

Deployments → 最新をリトライ(または `main` に何かプッシュされた時点で自動ビルド)。環境変数はビルド時に注入されるので、変数追加後は必ず再デプロイする。

✔ デプロイ後の公開 URL(`https://magical-journal.<アカウント>.workers.dev`)をブラウザで開き、画面が出ること(LIFF 外なので登録画面 or LIFF エラーになるが、白画面でなければ配信 OK)。
✔ この URL を **A-2 の `PUBLIC_ASSET_BASE_URL` に設定した値と一致**させる(違う場合は `.env` を直して A-3 の `up -d api` を再実行)。
✔ キャラ画像の配信確認: `https://magical-journal.<アカウント>.workers.dev/characters/03-kazema/male.webp` が画像として開けること(Flex カードの画像に必要)。

## C. 外部サービス設定

### C-1. Google Cloud(Maps キーのリファラー制限)

Google Cloud Console → 認証情報 → 対象 API キー → アプリケーションの制限「HTTP リファラー」に以下を追加:
- `https://magical-journal.<アカウント>.workers.dev/*`
- (開発用)`http://localhost:5173/*`

✔ 未設定だと本番でも地図・住所検索が `RefererNotAllowedMapError` で失敗する。API 制限で「Maps JavaScript API」「Geocoding API」が有効なことも確認。

### C-2. LINE Developers

**LINE Login チャネル → LIFF**:
- 対象 LIFF アプリのエンドポイント URL = フロント公開 URL(`https://magical-journal.<アカウント>.workers.dev`)

**Messaging API チャネル**:
- Webhook URL = `https://mj-api.iterra.online/line/webhook`
- Webhook の利用 = **オン**
- あいさつメッセージ・応答メッセージ(自動応答)= **オフ**(Bot の応答と競合させない)
- 「検証(Verify)」ボタンを押して **成功(200)** になること ✔

## D. 実 LLM の初回確認(バッチ)

**デプロイ前後で必ず1回**、実キーで運勢生成が通ることを手動確認する(既定モデル `claude-sonnet-5` / `gpt-5` が実在・利用可かの確認。エミュレーションでは代替不可)。

```
cd /volume1/docker/magical-journal/infra
# 登録済みユーザーが1人以上いる状態で:
sudo docker compose exec batch node dist/index.js run-daily
```
✔ ログに `total=N ok=N ng=0` が出て、LIFF の「今日のページ」に運勢文が表示されること。
✔ `ng` が出る場合はモデル名 or キーを疑う。`.env` に `CLAUDE_MODEL=` / `OPENAI_MODEL=` で正しいモデル ID を明示して `up -d batch` で再起動。

> 月次も同様に一度確認:`sudo docker compose exec batch node dist/index.js run-monthly`
> 以降は cron(日次 03:00 / 月次 毎月1日 03:30 JST)で自動実行される。

## E. 通し動作確認(実機)

LINE アプリの本人アカウント(`ALLOWED_LINE_USER_IDS` に入れた ID)で:

1. リッチメニュー or LIFF URL から起動 → 未登録なら登録画面 → 生年月日・氏名・住所(郵便番号検索)・キャラ選択で登録 ✔
2. マイタイプ画面に診断結果(キャラ画像・タイプ名・星座・気学・数秘)が表示 ✔
3. 今日のページに運勢文(D で生成済み)・方位グリッド・方位マップ ✔
4. トークで「マイタイプ」送信 → Flex タイプカードが返る ✔
5. トークで「今日の運勢」送信 → Flex 運勢カードが返る ✔
6. 設定で住所変更 → 方位マップの中心が変わる ✔

> 詳細な実機チェックは `docs/09_実機テストチェックリスト.md`(iOS/Android 各項目)を使う。

## 運用コマンド集(NAS)

```
# コード更新 → 再ビルド → 再起動(2回目以降の定番)
curl -L https://github.com/biz-iterra/magical-journal/archive/refs/heads/main.tar.gz | tar xz --strip-components=1 -C /volume1/docker/magical-journal
cd /volume1/docker/magical-journal/infra && sudo docker compose build api batch && sudo docker compose up -d api batch

# ログ
sudo docker compose logs -f api
sudo docker compose logs --tail 50 batch

# バッチ手動実行
sudo docker compose exec batch node dist/index.js run-daily [--date YYYY-MM-DD]
sudo docker compose exec batch node dist/index.js run-monthly [--force]

# 招待者を追加(.env の ALLOWED_LINE_USER_IDS にカンマ区切りで追記して再起動)
vi .env && sudo docker compose up -d api batch

# DB バックアップ(SQLite ボリューム)
sudo docker compose exec api sh -c 'cp /data/mj.sqlite /data/mj.backup.sqlite'
```

> キャラ側リポジトリを更新したとき: 画像・ペルソナの再取り込みは開発環境で `node scripts/import-characters.mjs` / `node scripts/import-personas.mjs` を実行してコミット → NAS を更新。

## トラブル早見表

| 症状 | 見るところ |
|------|-----------|
| `https://mj-api.iterra.online/api/health` が開かない | tunnel ログ / Cloudflare Public Hostname(`http://api:3000`)/ DNS が Cloudflare Active か |
| 全ユーザーが 403(`MJ-AUTH-004`) | `ALLOWED_LINE_USER_IDS` が空でないか |
| 地図・住所検索が出ない | `VITE_GOOGLE_MAPS_API_KEY` と Google のリファラー制限(`RefererNotAllowedMapError`) |
| 運勢文が出ない | バッチが走ったか(`logs batch`)、`ng` の有無、LLM キー・モデル名 |
| Flex カードに画像が出ない | `PUBLIC_ASSET_BASE_URL` と画像 URL が公開されているか |
| Webhook の Verify が失敗 | `LINE_CHANNEL_SECRET` が Messaging API チャネルのものか、Webhook URL が `/line/webhook` か |
| エラーコード（MJ-XXX）が出た | `docs/12_APIリファレンス・エラーコード.md` の対処法 |
