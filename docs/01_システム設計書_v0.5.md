# マジカルジャーナル システム設計書 v0.5

登録情報(生年月日・氏名・住所)を基にした診断群と、気学に基づく日次・月次の方位/運勢提示を行う LINE(LIFF)アプリのシステム設計。診断は**プラグイン型の診断モジュール**(docs/11)として実装し、コア改修なしで追加・保留できる。現在の有効モジュール: 星座 / ポテンシャルタイプ(キャラ表現) / 気学(プロファイル・方位) / 数秘術(ライフパス・ディスティニー)。姓名判断は**保留**。

> **v0.5 変更点**: 診断を拡張前提のモジュールアーキテクチャに再設計(docs/11 診断モジュール規約を新設)。姓名判断を保留(held)へ変更 — 仕様(docs/10)は保持、漢字氏名の収集・画数/吉凶マスタ整備は保留解除まで行わない。diag_results をモジュール汎用スキーマに変更。
> **v0.4 変更点**: プロファイル診断に数秘術(ライフパスナンバー・ディスティニーナンバー)と姓名判断(五格)を追加(仕様は docs/10)。登録フローに氏名入力を追加、画数・吉凶マスタと未確定論点を追加。
> **v0.3 変更点**: リッチメニュー改訂(方位マップを今日のジャーナルに統合し「友達のタイプ診断」を新設)、メニューごとの LLM 利用有無を明記、マイタイプを固定コピーの Flex カード化、キャラ整合のためのタイプ記述調整(3タイプ)を確定・別紙化。
> **v0.2 変更点**: キャラクターの男女2バリアント対応(表示スタイル選択方式)、キャラアセット取り込みパイプライン追加、リッチメニューの具体設計追加。

## 1. 設計方針

| 方針 | 内容 |
|------|------|
| コスト最小 | ランニングコストは実質 LLM API 利用料のみ(月数百円規模)を目標 |
| NAS 活用 | バックエンド一式を UGREEN NASync DXP4800 GT 上の Docker で稼働 |
| ロックイン回避 | 診断エンジンは純粋な TypeScript モジュールとして分離。LLM・地図・ホスティングはすべて差し替え可能な抽象化層を挟む |
| 決定的ロジックと生成の分離 | 診断・暦・方位判定はすべてコードで決定的に計算。LLM は文章表現のみ担当 |
| クローズド前提 | 招待制(LINE ユーザー ID の許可リスト)。スケールは後で考える |

## 2. 全体アーキテクチャ

```
[ユーザー(LINE アプリ)]
   │  リッチメニュー → LIFF 起動
   ▼
[LIFF フロントエンド]  … Cloudflare Pages(静的 SPA・無料)
   │  HTTPS / LIFF ID トークン
   ▼
[Cloudflare Tunnel]    … NAS のポート開放ゼロで公開(無料)
   ▼
┌─ NAS(UGREEN DXP4800 GT / Docker Compose)──────┐
│  api      : Node.js (Hono) REST API                │
│  batch    : node-cron 夜間バッチ                    │
│  db       : SQLite(単一ファイル、NAS スナップショットでバックアップ)│
│  暦データ : 節入り・年/月/日盤の静的マスタ           │
└────────────────────────────────┘
   │                          │
   ▼                          ▼
[LLM API]                [外部 API]
 Claude / GPT             LINE Messaging API(応答のみ=無料枠)
 (差し替え可能)           Google Maps JS / Geocoding(無料枠内)
```

### 各層の選定理由

- **フロントを Cloudflare Pages に分離**: 静的アセットは NAS に置かず CDN 配信。表示速度(当初の懸念)を確保し、NAS 停止時も UI 自体は落ちない。ITERRA の既存 Cloudflare 運用・`@iterra-inc/ui` と整合
- **Cloudflare Tunnel**: 自宅 NAS を直接インターネットに晒さない。ポート開放・固定 IP・リバースプロキシ設定が不要で、無料
- **SQLite**: クローズド(想定 100 ユーザー未満)では PostgreSQL は過剰。単一ファイルでバックアップ・移設が容易。将来クラウド移行時も Docker Compose ごと持ち出せる
- **GAS を採用しない理由**: GAS + Sheets でも構築可能だが、Web アプリからの同期呼び出しはコールドスタートで数秒かかることがあり、レスポンス懸念に反する。今回は NAS 常駐 API を採用(バッチの一部を将来 GAS に寄せることは可能)

## 3. フロントエンド(LIFF)

| 項目 | 内容 |
|------|------|
| フレームワーク | React + Vite + TypeScript |
| スタイル | vanilla-extract(`@iterra-inc/ui` の資産・知見を流用) |
| 認証 | LIFF ID トークン → API 側で LINE の検証エンドポイントにより検証 → セッション発行 |
| 地図 | Google Maps JavaScript API |

### 画面構成

1. **初回登録**: 入力項目は**有効な診断モジュールの要求入力から自動導出**する(docs/11 §2)。現時点では: 生年月日(西暦のみ・ピッカーで強制/+任意で出生時刻)、氏名かな(→ ヘボン式ローマ字を自動変換しユーザーが確認・確定)、自宅住所(Geocoding は登録時 1 回)。**漢字氏名は保留中の姓名判断専用のため収集しない**。最後に**キャラ表示スタイルの選択**(下記)。将来モジュールが新入力を要求する場合は「追加入力で新診断が解放」導線で追加収集する
2. **マイタイプ(詳細)**: enabled モジュールの結果をセクションとして動的に描画する(docs/11 §5)。現在: 星座・ポテンシャルタイプ(キャラ)・気学プロファイル・ライフパス・ディスティニー。説明文は固定コピー(LLM 不使用)。モジュール追加時に画面改修は不要。ポテンシャルタイプはキャラクターで表現。ハイブリッドの場合は主キャラ(大)+副キャラ(小)を並列表示。リッチメニュー④の Flex カードの「詳しく見る」の遷移先
3. **今日のページ(ジャーナル本体)**: 今日の運勢文(事前生成済み)、吉方位・凶方位、**方位マップを同一画面に統合**して表示
4. **友達のタイプ診断**: `clientSafe` な enabled モジュールのみを対象に表示する(docs/11 の一覧参照)。生年月日で星座・タイプ+キャラ・本命星・ライフパス、氏名かなを任意入力すればディスティニーも表示。**計算は LIFF 内(端末側の診断エンジン)で完結させ、入力値をサーバーへ送信・保存しない**。方位・運勢は本人の住所が前提のため対象外とし、性格診断のみに限定する
5. **月間ページ**: 月盤ベースの月方位・月運
6. **設定**: 住所修正(修正時のみ Geocoding 再実行)、出生時刻の追記、キャラ表示スタイルの切り替え

### キャラ表示スタイル(男女バリアントの扱い)

各キャラクターには男女 2 バリアントが存在する(例: IL+ = カゼマ/カザネ)。設計は以下とする。

- ユーザーの**性別は収集しない**。診断ロジックに性別は不要であり、収集する個人情報は最小限に留める
- 代わりに初回登録時、自分のタイプの男女キャラを並べて提示し「どちらのキャラで表示するか」を**ユーザー自身が選択**する(設定でいつでも変更可)
- 選択値は `profiles.char_style`(male / female)として保持し、ハイブリッドの主・副キャラ両方に同じスタイルを適用する
- 運勢文生成時のトーンも選択バリアントの一人称・口調定義を使用する

### 方位マップ表示

- 自宅座標を中心に、8方位の扇形ポリゴンをオーバーレイ描画
- 四正(北・東・南・西)= 30度、四隅(北東など)= 60度
- 吉方位 = 緑系、凶方位 = 赤系、中立 = 無色の塗り分け
- 距離リング(例: 10km / 50km / 100km)を補助線として描画
- 地図コンポーネントは抽象化し、Leaflet + 地理院タイル(完全無料)への差し替えを可能にしておく

## 4. バックエンド(NAS / Docker)

### コンテナ構成(docker-compose)

| サービス | 役割 |
|----------|------|
| `api` | REST API。診断計算、当日データ返却、住所更新、LIFF トークン検証 |
| `batch` | 夜間バッチ(節切り翌日分の方位判定+LLM 文章生成)、LINE 応答 Webhook 処理 |
| `tunnel` | cloudflared。Tunnel 経由で api を公開 |

DB は SQLite ファイルを `api`/`batch` で共有(WAL モード)。

### 診断エンジン(コアモジュール)

UI・API から独立した純 TypeScript パッケージとして実装し、ユニットテストを必須とする。

- `zodiac.ts` : 星座判定(日付範囲テーブル)
- `potential.ts` : ポテンシャル算出値(基準日 1920-01-01、mod 60)→ 対応表 → ハイブリッド判定(23:00〜23:59)
- `kigaku/honmei.ts` : 本命星・月命星(立春・節入り基準)
- `kigaku/ban.ts` : 年盤・月盤・日盤の中宮決定と 8 方位への星配置
- `kigaku/direction.ts` : 五黄殺・暗剣殺・歳破/月破/日破・本命殺・本命的殺・吉方位の判定
- `numerology.ts` : ライフパスナンバー・ディスティニーナンバー(docs/10。マスターナンバー・ローマ字規則含む)
- `mapping.ts` : ポテンシャルタイプ ↔ キャラクターの対応表参照
- `registry.ts` : 診断モジュールレジストリ(docs/11 のインターフェース実装)。各診断は DiagnosisModule として登録し、登録フロー・マイタイプ・バッチはレジストリ経由でのみ診断へアクセスする
- (保留)`seimei.ts` : 姓名判断。保留解除(docs/10 §6 #4, #6 の確定)まで実装しない

### 夜間バッチのフロー(毎日 03:00)

1. 暦マスタから当日の年盤・月盤・日盤を取得
2. アクティブユーザーごとに吉方位・凶方位・運勢の構造化データを算出
3. LLM に構造化データを渡し、当日の運勢文・過ごし方提案を生成(プロンプトにキャラのトーン定義を注入)
4. `daily_fortunes` に保存 → ユーザーが LIFF を開いた瞬間に表示(体感遅延ゼロ)

LLM 呼び出しは `LlmProvider` インターフェースで抽象化し、Claude / GPT を設定で切り替え可能にする。

### キャラアセット取り込みパイプライン

キャラクターの一次情報(YAML 定義・イメージ画像・カラーコード)は別リポジトリ(キャラクター制作側の Claude Code プロジェクト、`characters/NN-name/` 構成)にある。本アプリはこれを**参照先(source of truth)**とし、二重管理しない。

```
characters リポジトリ(YAML + 画像)
   │  import スクリプト(ビルド時 or 手動実行)
   ├─ YAML(名前・口調・カラー・正確性定義)→ characters / character_variants テーブルへ投入
   └─ 画像 → LIFF フロントの静的アセットへコピー(Cloudflare Pages にデプロイ)
```

- import スクリプトは冪等(再実行で上書き更新)とし、キャラ側リポジトリの更新をコマンド 1 回で反映できるようにする
- マッピング表 CSV の `定義ファイル(参照)`列がリポジトリ内ディレクトリへの対応キーとなる
- LLM プロンプトに注入するトーン定義は YAML から直接生成する(診断文テンプレート設計時に注入項目を確定)

## 5. データ設計(主要テーブル)

| テーブル | 主なカラム | 備考 |
|----------|-----------|------|
| `users` | id, line_user_id, display_name, is_allowed | 許可リスト方式 |
| `profiles` | user_id, birth_date, birth_time, name_kana, name_romaji, address_text, lat, lng, char_style | name_romaji はユーザー確定値。漢字氏名カラムは保留解除時に追加。char_style = キャラ表示スタイル(male/female) |
| `diag_results` | user_id, module_id, module_version, result_json, computed_at | **モジュールごとに 1 行**の汎用スキーマ(docs/11 §4)。固定カラムを増やさない |
| `module_config` | module_id, status, display_order | enabled / held / planned のステータス管理 |
| `master_module_contents` | module_id, key, text | 各モジュールの固定コピー(数秘の数意味など) |
| `characters` | potential_type, motif, theme_color, accent_color, asset_dir, catch_copy | タイプ単位の共通情報。asset_dir = キャラリポジトリ内ディレクトリ |
| `character_variants` | potential_type, gender, name, first_person, tone, image_ref | 男女バリアント(12タイプ × 2 = 24 行)。YAML から import |
| `cal_terms` | year, month_index, term_name, term_datetime | 節入りマスタ |
| `cal_year_ban` | kigaku_year, star, eto | 年盤 |
| `cal_month_ban` | kigaku_year, month_index, star | 月盤 |
| `cal_day_ban` | date, star, tonpu_mode, eto | 日盤(陽遁/陰遁含む) |
| (保留)`master_kanji_strokes` / `master_seimei_fortune` | — | 姓名判断用マスタ。保留解除時に整備 |
| `daily_fortunes` | user_id, date, directions_json, fortune_text | バッチ生成結果 |

### 暦マスタの整備(Phase 0 のブロッカー)

- 対象範囲: 節入り・年盤・月盤 = **1920〜2050年**(生年月日入力の全域をカバー)、日盤 = **当面 2025〜2036年**
- 生成方法: 節入りは天文計算式(または理科年表由来データ)からスクリプト生成。日盤の陽遁陰遁切替は採用流派のルールで生成
- **検証手順(必須)**: Yuki さんが信頼する気学暦(書籍・市販の万年暦)を基準として指定し、生成データからサンプル日(切替日付近を重点的に)を突合。基準暦の指定が Phase 0 の前提条件

### 個人情報の扱い

生年月日+住所は個人情報に該当する。クローズドでも以下を実施:

- SQLite ファイルの保存領域は NAS の暗号化ボリューム上に配置
- バックアップは NAS スナップショット+既存のコールドアーカイブ方針に組み込み
- API は許可リスト外の LINE ユーザー ID を登録段階で拒否
- **友達診断で入力される第三者の生年月日は、端末内(LIFF の診断エンジン)で処理を完結させ、サーバーへの送信・保存を一切行わない**(第三者の個人情報を預からない設計)

## 6. LINE 設計

| 項目 | 設計 |
|------|------|
| 導線 | リッチメニュー → LIFF 起動(通数課金なし) |
| プッシュ通知 | **日次プッシュは行わない**。無料プラン(月 200 通)では日次配信はユーザー数 6 人で枯渇するため |
| 通知の代替 | ユーザー発のメッセージへの**応答メッセージは無料・無制限**。「おはよう」等に今日の運勢カードを返す応答型で、実質的な毎朝体験を提供 |
| プッシュの使い所 | 無料枠 200 通は月初の月運配信・重要なお知らせに限定して温存 |
| 将来 | 日次プッシュを本気でやる場合はライトプラン(月 5,000 円)を課金設計とセットで検討 |

### リッチメニュー設計

サイズ 2500×1686px、2行 × 3列の 6 分割。単一メニュー(切替なし)で v1 は運用する。

```
┌──────────────┬──────────────┬──────────────┐
│ ① 今日のジャーナル │ ② 友達のタイプ診断 │ ③ 今日の運勢を聞く │
├──────────────┼──────────────┼──────────────┤
│ ④ マイタイプ      │ ⑤ 月間運勢       │ ⑥ 設定           │
└──────────────┴──────────────┴──────────────┘
```

| # | メニュー | アクション | 挙動 | LLM 利用 |
|---|---------|-----------|------|---------|
| ① | 今日のジャーナル | LIFF 起動 | 今日のページ。運勢文+吉凶方位+**方位マップを 1 画面に統合**。メイン導線のため左上に配置 | あり(夜間バッチで事前生成した文章の表示のみ) |
| ② | 友達のタイプ診断 | LIFF 起動 | 生年月日入力 → 3軸タイプ+キャラを表示。計算は端末内で完結、入力値は送信・保存しない | なし(完全に決定的計算) |
| ③ | 今日の運勢を聞く | メッセージ送信 | 「今日の運勢」を送信 → Bot が応答メッセージ(無料)で Flex 運勢カードを返信。①の事前生成文の要約を再利用し、カードから①への LIFF リンクを張る | あり(事前生成の再利用。リアルタイム生成なし) |
| ④ | マイタイプ | メッセージ送信 | 「マイタイプ」を送信 → Bot が応答メッセージで Flex タイプカード(キャラ画像+タイプ名+固定コピー)を返信。「詳しく見る」でマイタイプ詳細 LIFF へ | **なし**(12タイプ・星座・本命星は確定情報のため固定コピー) |
| ⑤ | 月間運勢 | LIFF 起動 | 月間ページ | あり(月次バッチで事前生成) |
| ⑥ | 設定 | LIFF 起動 | 住所・出生時刻・キャラ表示スタイルの変更 | なし |

- **LLM のリアルタイム呼び出しはゼロ**。文章生成はすべて夜間(日次)・月初(月次)のバッチに閉じ、ユーザー操作時は保存済みデータの表示・返信のみ。レスポンスは全メニューで即時
- ③④が応答型設計(§6 冒頭)の入口。「LIFF を開くほどではないが軽く見たい」日常接点を無料で成立させる
- 未登録ユーザーがどのメニューを押しても、LIFF 側で登録画面へリダイレクトする(③④の場合は応答メッセージで登録用 LIFF リンクを案内)。②は未登録でも利用可とし、登録前の体験導線を兼ねる
- メニュー画像のデザインはキャラのテーマカラー体系と合わせて Phase 3 で制作

## 9. 設計上の論点・残課題

1. **基準暦の指定(Phase 0 ブロッカー)**: 日盤検証の基準にする気学暦(流派)を Yuki さんが指定する
2. **方位の北基準**: 真北か磁北か(東京では約 8 度西偏)。設定フラグとして実装し、デフォルトは要決定
3. **比和の扱い / 小児殺等の追加凶方位**: 吉凶判定ルールの最終確定(前回仕様書の論点表 #4, #6, #7)
4. **NAS 可用性**: 自宅停電・回線断でバッチと API が止まるリスク。クローズドでは許容とし、Compose 一式をクラウド(例: 低額 VPS)へ持ち出せる状態を維持することで対処
5. **診断文テンプレート**: LINE/LIFF 向けトーン定義は本設計確定後に別途提案(character-generator の YAML を LINE 向けに再設計)
6-a. **数秘術の方式**: ライフパス計算方式(全桁合算/年月日個別還元)、マスターナンバー採否、ローマ字長音規則。docs/10 に既定値を定義済み、人間の確認で確定する
6-b. **姓名判断は保留(held)**: 仕様(docs/10)は保持。保留解除の条件 = 画数の字体基準と吉凶表出典の指定(docs/10 §6 #4, #6)。解除時に漢字氏名の追加収集・マスタ整備・再計算バッチを docs/11 §6 の手順で実施する。Phase 0 のブロッカーからは除外
6. **キャラと診断タイプの性格差異(決定済み)**: 役割分離を採用。**診断の中身(強み・伸びしろ)はポテンシャルタイプ仕様を正、語り口・人格はキャラ YAML を正**とする。差異の大きい 3 タイプ(IR−・EL+・PR−)はユーザー向け表示文言のみキャラに整合させて調整済み(別紙「タイプ記述調整表」参照)。**算出ロジック・対応表は一切変更しない**
