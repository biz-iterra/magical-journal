## 絶対に守るルール

1. **決定的ロジックを LLM にやらせない。** 診断・暦・方位判定はすべてコードで計算する。LLM の役割は構造化データからの文章生成のみ
2. **診断ロジックの固定値を変更しない。** ポテンシャル基準日(1920-01-01)・周期(60)・対応表・ハイブリッド時間帯(23:00〜23:59)、気学の計算式、数秘の変換表・還元ルール、姓名判断の五格定義は仕様書で固定。画数辞書・吉凶表もマスタデータであり実行時に生成・推測しない。変更すると全ユーザーの診断結果が変わる
3. **暦は必ずマスタテーブルを参照する。** 節入り・年盤・月盤・日盤を実行時に天文計算しない。マスタ生成は `packages/calendar-data` のスクリプトに閉じる
4. **シークレットのハードコード禁止。** LINE チャネルシークレット・LLM API キー・Maps API キーは `.env`(gitignore 済み)+ Docker の env 経由。コード・コミットに含めない
5. **第三者情報を預からない。** 友達診断の入力値はサーバーへ送信・保存しない(LIFF 内で計算完結)
6. **LLM のリアルタイム呼び出しをしない。** 文章生成は夜間・月次バッチのみ。ユーザー操作時は保存済みデータの表示・返信のみ
7. **タイプの表示文言は調整表が正。** IR−/EL+/PR− のタイプ名・強み・伸びしろは `docs/04` の新文言を使用する
8. **診断の追加・変更は docs/11 のモジュール規約に従う。** レジストリを経由しない診断実装、保留(held)モジュールの本番有効化、held モジュール専用入力(漢字氏名)の収集をしない。登録フローの入力項目は enabled モジュールの要求から導出する

## リポジトリ構成

```
magical-journal/
├─ CLAUDE.md
├─ docs/                  # 仕様書(一次情報)
├─ packages/
│  ├─ engine/             # 診断エンジン(純TS・依存ゼロ・LIFFとAPIの両方から利用)
```

pnpm workspace のモノレポ。`engine` はブラウザ(LIFF)と Node(api/batch)の両方で動くこと(DOM/Node API に依存しない)。

## テスト方針

`packages/engine` はユニットテスト必須(vitest)。以下の検算例は最低限テストに含める:

- 気学 本命星: 1988年生(立春後)→ 三碧(3)、1990年生 → 一白(1)、2026年 → 一白中宮
- 数秘 ライフパス: 1990-05-17 → 合計32 → 5 / 合計が29になる生年月日 → 11(マスターナンバーで停止)/ マスターナンバー還元フラグ OFF 時は 11→2
- (保留解除時)姓名判断: 一字姓・一字名の霊数(総格には加えない)、画数82以上の 81 数理循環、辞書外漢字の手入力画数の反映
- モジュールレジストリ: enabled モジュールのみが登録フロー入力要求・マイタイプ表示・友達診断(clientSafe のみ)に現れること
- 気学 立春境界: 立春前日の生年月日が前年扱いになること
- 盤の配置: 中宮=五黄(5)のとき後天定位盤に一致すること
- ポテンシャル算出値: 1920-01-01 → 経過日数0 → 算出値1 → IR+ / 1920-01-02 → 算出値2 → ER+ / 1920-03-01 → 経過日数60(うるう年)→ 算出値1 → IR+
- ハイブリッド: 23:00 生まれで当日+翌日、22:59 で単一。当日と翌日が同タイプ(例: 算出値27→28)なら単一
- 対応表: 12タイプの割当数が 6件×6タイプ・4件×6タイプであること(仕様の分布と全数一致)
- 方位判定: 五黄中宮のとき五黄殺・暗剣殺が発生しないこと、本命星中宮のとき本命殺が発生しないこと

暦マスタは生成後、yakumoin.info の日盤データとの突合検証に合格するまで Phase 1 に進まない。

## コーディング規約

- ハードコード禁止・設定は config に分離(既存の GAS 規約と同思想)
- `engine` の関数は純粋関数とし、`Date` の直接生成を避けて日付文字列/数値で受け渡す(タイムゾーン事故防止。暦は JST 前提)
- エラーは握りつぶさない。バッチは失敗ユーザーをスキップして続行し、失敗一覧をログに残す
- コミットは機能単位で日本語メッセージ可
